# A1. Images

## A1-1. Informative Images

### Question
> _in the information simple or complex ?_

### Options
- Simple Informative Images (can be described briefly in alt text)
- Complex Images (requires detailed description: charts, diagrams, infographics, maps)

---

## A1-1-1. Simple Informative Images

### Checklist

- Add descriptive `alt` text that conveys the image’s meaning or purpose.
- Keep `alt` text concise. A short replacement is usually enough if the image is simple.
- Do not start with words like “image of”, “picture of”, or “graphic of”
- If the image contains text that is essential to understanding it, include that text in the `alt`.
- The `alt` text should provide the same information or equivalent purpose as the image.
- If surrounding text already fully explains the same information, keep the `alt` short and avoid repeating the full explanation.

### Code
```html
<img src="graph.png" alt="Sales increased 25% from Q1 to Q2 in 2024">
```

### Learn More  
- Non-text content requirement: [WCAG 1.1.1 - Non-text Content](https://www.w3.org/WAI/WCAG22/quickref/#non-text-content)
- Image alt attribute: [MDN - img alt attribute](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/img#alt)
- Informative images tutorial: [W3C WAI - Informative Images](https://www.w3.org/WAI/tutorials/images/informative/)
- Alternative text techniques: [WebAIM - Alt Text](https://webaim.org/techniques/alttext/) 
  
---  
  
## A1-1-2. Complex Images  
  
### Checklist  
- Provide short `alt` text that identifies the image type or topic.
- Also provide a full text alternative nearby, on the same page, or on a clearly linked separate page.
- The full description should include all important data points, labels, trends, relationships, and conclusions needed to understand the image.
- If the image is a chart or graph, describe the main trend, key comparisons, and notable highs/lows.
- Do not rely on color alone to communicate meaning.
- Where lines, bars, points, boundaries, or other essential graphical parts must be visually distinguished, they should have at least **3:1 contrast** against adjacent colors or the background.
- If the same data can be presented as a table, consider providing the data table in addition to the image description.
- If there is a visible description on the page that fully explains the image, keep the `alt` short and let the nearby text carry the detail.
  
### Code  
```html
<img
  src="sales-chart.png"
  alt="Bar chart showing quarterly sales growth"
  aria-describedby="chart-description"
>

<div id="chart-description">
  Quarterly sales for 2024: Q1 $50,000, Q2 $65,000, Q3 $72,000, Q4 $90,000.
  Sales increased each quarter, with the highest value in Q4. Overall growth from
  Q1 to Q4 was 80%.
</div>
```

### Learn More  
- Complex images tutorial: [W3C WAI - Complex Images](https://www.w3.org/WAI/tutorials/images/complex/)
- Non-text content requirement: [WCAG 2.2 - Non-text Content (1.1.1)](https://www.w3.org/WAI/WCAG22/quickref/#non-text-content) 
- Charts and graphs accessibility: [W3C WAI - Charts and Graphs](https://www.w3.org/WAI/tutorials/images/complex/#charts-and-graphs)  
  
---  
  
## A1-2. Decorative Images  
  
### Checklist  
- Use an empty `alt` attribute: `alt=""`.
- Decorative images should not add information, meaning, or functionality.
- If an image is purely decorative and placed with HTML, use `alt=""`.
- If the image is only visual styling, CSS background images are often a better choice.
- Do not include decorative images in the reading order for assistive technologies.
- Do not use meaningful filenames or surrounding text as a substitute for proper handling.
- Decorative dividers, patterns, flourishes, and purely visual illustrations should follow this approach.
- Do not add unnecessary ARIA to a plain decorative `<img>` when `alt=""` already handles it.
  
### Code  
```html
<!-- Decorative image -->
<img src="decorative-border.png" alt="">

<!-- Decorative background -->
<div class="section-divider"></div>
```

```css
.section-divider {  
background-image: url("pattern.png");  
}
```
### Learn More  
- Decorative images tutorial: [W3C WAI - Decorative Images](https://www.w3.org/WAI/tutorials/images/decorative/)

## A1-3-1. Image Buttons / Controls  
  
### Checklist  
- The accessible name should describe the **action**, not the image appearance.
- Be specific, for example: “Search”, “Submit form”, “Open menu”, “Close dialog”.
- Do not use vague labels like “button” or “icon”.
- If the button already contains visible text, make the image decorative with `alt=""`.
- Use a real `<button>` for actions.
- Ensure the control is keyboard accessible.
- Make sure the button’s visible focus indicator is clearly visible.
- Pointer targets should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
  
### Code  
```html
<!-- Image-only button -->  
<button type="submit" aria-label="Search">  
<img src="search-icon.svg" alt="">  
</button>  
  
<!-- Button with image and visible text -->  
<button type="submit">  
<img src="search-icon.svg" alt="">  
Search  
</button>
```
  
### Learn More  
- Accessible icon buttons: [Sara Soueidan - Accessible Icon Buttons](https://www.sarasoueidan.com/blog/accessible-icon-buttons/)
  
---  
  
## A1-3-2. Image Links  

### Checklist  
- The accessible name should describe the **destination** or purpose of the link.    
- Be specific about where the link goes, for example: “View products”, “Go to homepage”.
- If the link contains both an image and visible text, make the image decorative with `alt=""`.
- Use meaningful link text. Do not rely on the image alone unless the image has an appropriate text alternative.
- Ensure the link is keyboard accessible.
- If the link is icon-only, provide an accessible name that describes the destination.
- Pointer targets should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
  
### Code  
```html
<!-- Image-only link -->
<a href="/home">
  <img src="logo.png" alt="Go to homepage">
</a>

<!-- Link with image and text -->
<a href="/products">
  <img src="products-icon.png" alt="">
  View products
</a>
```
  
### Learn More  
- Functional images tutorial: [W3C WAI - Functional Images](https://www.w3.org/WAI/tutorials/images/functional/)
- Link purpose requirement: [WCAG 2.4.4 - Link Purpose (In Context)](https://www.w3.org/WAI/WCAG22/quickref/#link-purpose-in-context)
  
---  
  
## A1-3-3. Icons with Visible Text  

### Checklist  
- The icon should usually be decorative when the visible text already gives the accessible name.
- Keep the visible text clear and specific.
- Do not duplicate the same information in both the icon `alt` and the visible text.
- Ensure the text label is part of the interactive element.
- Ensure the control is keyboard accessible.
- If the icon communicates additional meaning beyond the visible text, that meaning must also be available in text.
- Essential visual indicators used as part of the control should have at least **3:1 contrast** against adjacent colors.
  
### Code  
```html
<a href="/settings">  
<img src="settings-icon.svg" alt="">  
Settings  
</a>
```
  
### Learn More  
- Accessible icon buttons: [Sara Soueidan - Accessible Icon Buttons](https://www.sarasoueidan.com/blog/accessible-icon-buttons/)
  
---  
  
## A1-3-4. Standalone Icon Controls  

### Checklist  
- Provide an accessible name using one method only, such as visible text, `aria-label`, or `aria-labelledby`.
- Use a native interactive element like `<button>` or `<a>` when the icon is clickable.
- If the icon itself is decorative inside the control, hide it from assistive tech with `aria-hidden="true"`.
- Do not add `role="button"` to a real `<button>`.
- Do not add unnecessary ARIA when visible text already provides the accessible name.
- Ensure the control is keyboard accessible.
- Pointer targets should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
- Essential icon graphics that users must perceive should have at least **3:1 contrast** against adjacent colors.
### Code  
```html
<!-- Standalone icon button -->  
<button aria-label="Close dialog">  
<svg aria-hidden="true" viewBox="0 0 24 24" focusable="false">  
<use href="#close-icon"></use>  
</svg>  
</button>
```
  
### Learn More  
- Accessible icon buttons: [Sara Soueidan - Accessible Icon Buttons](https://www.sarasoueidan.com/blog/accessible-icon-buttons/)
- Name, role, value requirement: [WCAG 4.1.2 - Name, Role, Value](https://www.w3.org/WAI/WCAG22/quickref/#name-role-value)

## A2-1-1. Prerecorded Video with Audio  

### Checklist  
- Provide synchronized captions for all spoken dialogue and essential non-speech audio.
- Captions should include speaker identification when needed for understanding.
- Captions should include meaningful sound information when needed for understanding, such as alarms, laughter, applause, or doors slamming.
- Provide audio description for important visual information that is not conveyed in the audio, or provide a full descriptive transcript as an alternative where appropriate.
- Ensure the media player is keyboard accessible.
- Ensure all player controls have accessible names.
- Do not autoplay unless it is essential; if audio plays automatically for more than 3 seconds, users need a way to pause, stop, or control volume.
- Provide pause and stop controls that are easy to find and use.
- Make sure controls and essential icons have at least **3:1 contrast** against adjacent colors.
- Text in controls should meet **4.5:1 contrast** if normal-sized, or **3:1** if large text.
- Pointer targets for player controls should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
  
### Code  
```html
<video controls>  
<source src="video.mp4" type="video/mp4">  
<track kind="captions" src="captions.vtt" srclang="en" label="English captions" default>  
<track kind="descriptions" src="descriptions.vtt" srclang="en" label="English descriptions">  
Your browser doesn't support video.  
</video>  
  
<details>  
<summary>Read transcript</summary>  
<p>[Full transcript of video content]</p>  
</details>
```

# B1. Text Input Fields

### Options

- Password or sensitive data
- General text input
- Numbers or specific formats

---

## B1-1. Personal Information Inputs

### Checklist
- Every input has a visible label element.
- Label is associated using `for` attribute matching the input’s `id`.
- Use appropriate `autocomplete` attribute for the data type.
- Mark required fields with the `required` attribute.
- Indicate required fields in text, not by color alone.
- Use the appropriate input type where available, for example `email`, `tel`, or `url`.
- Provide clear format instructions before the field when a specific format is required.
- Placeholder text must not replace the label.
- If an input error is shown, the error text should be programmatically associated with the field.
- Label text and form text should meet at least **4.5:1 contrast** against the background.
- Input boundaries and focus indicators should be clearly visible; visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- Pointer targets for controls should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<label for="fullname">Full Name *</label>
<input
  type="text"
  id="fullname"
  name="fullname"
  autocomplete="name"
  required>

<label for="email">Email Address *</label>
<input
  type="email"
  id="email"
  name="email"
  autocomplete="email"
  required>

<label for="phone">Phone Number</label>
<input
  type="tel"
  id="phone"
  name="phone"
  autocomplete="tel">
```

## B1-2. Password Inputs

### Checklist

- The slider must be keyboard accessible; arrow keys should adjust the value.
- Provide minimum and maximum values in text or through surrounding context when needed.
- Provide the step increment when it matters to the task.

# C1. Text Links

### Question  
> _Does the link text clearly describe where it goes or what it does?_  

### Options  
- Yes, the link text is descriptive (e.g., "View our pricing plans")
- No, it uses generic text (e.g., "Click here", "Read more", "Learn more")
- It's an image link
- It's an icon link

---

## C1-1. Descriptive Text Links

### Checklist
- Link text clearly describes the destination or purpose.
- Link text should make sense out of context, not only with surrounding text.
- Do not use visually distinguishable styling as the only way to make links understandable.
- Do not rely on color alone to indicate links.
- Underline text links or provide another strong non-color indicator.
- Link text should meet at least **4.5:1 contrast** against the background.
- The visual distinction between link text and surrounding body text should use more than color alone; if color is the only difference, the link color needs at least **3:1 contrast** against surrounding text.
- Provide a clearly visible focus indicator when navigating by keyboard.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- Links must be keyboard accessible.
- Visited and unvisited links should remain distinguishable if different states are shown.
- Pointer targets for links should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<p>Check out our <a href="/pricing">pricing plans</a>.</p>
<p><a href="/contact">Contact our support team</a>.</p>
<p>Learn more about <a href="/accessibility">web accessibility best practices</a>.</p>
```

### Learn More
- Link purpose in context: [WCAG 2.4.4 - Link Purpose (In Context)](https://www.w3.org/WAI/WCAG22/quickref/#link-purpose-in-context)
- Distinguishing links from text: [WCAG 1.4.1 - Use of Color](https://www.w3.org/WAI/WCAG22/quickref/#use-of-color)
- Contrast requirements for links: [WCAG 1.4.3 - Contrast (Minimum)](https://www.w3.org/WAI/WCAG22/quickref/#contrast-minimum)
- Focus visibility: [WCAG 2.4.7 - Focus Visible](https://www.w3.org/WAI/WCAG22/quickref/#focus-visible)

## C1-2. Improving Generic Links

### Checklist
- Rewrite generic link text so it describes the destination or action.
- If the visible text cannot be changed, add hidden context inside the link.
- If needed, connect the link to nearby descriptive text using `aria-describedby`.
- Avoid repeated generic labels like "Click here", "Read more", or "Learn more" without context.
- Where multiple similar links appear on one page, make each accessible name unique.
- Link text should meet at least **4.5:1 contrast** against the background.
- If color alone differentiates links from surrounding text, link color should have at least **3:1 contrast** against surrounding text.
- Provide a clearly visible keyboard focus indicator with at least **3:1 contrast** against adjacent colors.
- Pointer targets for links should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<!-- Better rewritten link -->
<p>View our affordable pricing. <a href="/pricing">View pricing plans</a>.</p>

<!-- Hidden context inside the link -->

  <h2>Introduction to Web Accessibility</h2>
  <a href="/articles/accessibility-intro">
    Read more<span class="visually-hidden"> about Introduction to Web Accessibility</span>
  </a>
</article>

<!-- Using aria-describedby -->
<article>
  <h2 id="article1">How to improve your usability</h2>
  <a href="/articles/form-tips" aria-describedby="article1">Read article</a>
</article>
```

# C3. Skip Links

## C3. Skip Links

### Checklist
- The skip link should be the first focusable interactive element on the page.
- It should become visible when it receives keyboard focus.
- Link text should clearly describe the destination, for example "Skip to main content".
- The target must have a valid `id`.
- The target should be a valid landmark or heading near the start of the destination content.
- Skip links should bypass repeated content.
- Skip links must work with keyboard navigation.
- Visible skip link text should meet at least **4.5:1 contrast** against the background.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- Pointer targets for links should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<body>
  <a href="#main" class="skip-link">Skip to main content</a>

  <header>
    <nav aria-label="Main navigation">
      <!-- Navigation items -->
    </nav>
  </header>

  <main id="main">
    <h1>Page Title</h1>
  </main>
</body>
```

### CSS
```css
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: #fff;
  padding: 8px;
  text-decoration: none;
  z-index: 100;
}

.skip-link:focus {
  top: 0;
}
```

# C4. Breadcrumbs

## C4. Breadcrumbs

### Checklist
- Wrap breadcrumbs in a `<nav>` element with an accessible name such as `aria-label="Breadcrumb"`.
- Use an ordered list for breadcrumb items.
- Add visual separators with CSS or markup that does not create redundant announcements.
- The current page should be the last item in the trail.
- The current page should not be a link, or it should be marked with `aria-current="page"`.
- All breadcrumb links must be keyboard accessible.
- Breadcrumb text should meet at least **4.5:1 contrast** against the background.
- If separators or current-location indicators are needed visually, they should have at least **3:1 contrast** against adjacent colors when relied on for understanding.
- Provide a clearly visible focus indicator when navigating by keyboard.
- Pointer targets for links should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

```html
<nav aria-label="Breadcrumb">

    <li><a href="/products">Products</a></li>
    <li><a href="/products/laptops">Laptops</a></li>
    <li aria-current="page">MacBook Pro</li>
  </ol>
</nav>
```

```css
nav[aria-label="Breadcrumb"] li + li::before {
```

# C5. Pagination

## C5. Pagination

### Checklist
- Wrap pagination in a `<nav>` element with an accessible name such as `aria-label="Pagination"`.
- Mark the current page with `aria-current="page"`.
- The current page should be clearly identified visually and programmatically.
- Previous and Next links should have clear labels.
- First and Last links can be included when helpful.
- All pagination links must be keyboard accessible.
- Provide a clearly visible focus indicator on all interactive items.
- Pagination links should have at least **24 × 24 CSS pixels** target size, unless a WCAG exception applies.
- Text labels should meet at least **4.5:1 contrast** against the background.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- If helpful, show the current result range or page range in text.

### Code
```html
<nav aria-label="Pagination">
  <ul>
    <li><a href="?page=1" aria-label="Go to previous page">Previous</a></li>
    <li><a href="?page=1">1</a></li>
    <li><span aria-current="page">2</span></li>
    <li><a href="?page=3">3</a></li>
    <li><a href="?page=3" aria-label="Go to next page">Next</a></li>
  </ul>
</nav>

<p>Showing 11–20 of 57 results</p>
```

# C6. Anchor Links

## C6. Anchor/Jump Links

### Checklist
- Link text should describe the destination section.
- The target element must have a valid `id`.
- The target should usually be a heading or the start of a region.
- Consider moving focus to the destination when that better supports the interaction.
- Links must be keyboard accessible.
- Provide a clearly visible focus indicator when navigating by keyboard.
- For long pages, consider adding a "Back to top" link.
- Test that hash links work correctly and do not break browser navigation.
- Visible link text should meet at least **4.5:1 contrast** against the background.
- If color alone differentiates links from surrounding text, link color should have at least **3:1 contrast** against surrounding text.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

# D1. Headings

### Question  
> _Is this the main page title or a section heading?_  

### Options  
- Main page title (H1)
- Section headings (H2-H6)

---

## D1-1. Page Title (H1)

### Checklist
- Every page should have exactly one main `<h1>` that identifies the page content.
- The `<h1>` should be the first main heading in the page content.
- The `<h1>` text should match or closely relate to the document `<title>`.
- Do not hide the real page heading with CSS if it is needed for structure and understanding.
- Do not skip directly from `<h1>` to `<h3>` or lower when the content hierarchy needs an `<h2>`.
- Use a real heading element, not a styled `<div>` or `<span>`.
- Heading text should be concise and descriptive.
- Heading text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** if it qualifies as large text.

### Code
```html
<head>
  <title>About Us - Company Name</title>
</head>
<body>
  <main>
    <h1>About Us</h1>
    <p>Content about the company...</p>
  </main>
</body>
```

## D1-2. Section Headings

### Checklist
- Never skip heading levels when a skipped level is needed for the content hierarchy.
- Heading levels should follow a logical outline.
- Each heading should describe the content that follows it.
- Use actual heading elements (`<h2>` to `<h6>`), not styled non-semantic elements.
- Do not choose heading level based only on visual size.
- Multiple headings at the same level are fine when they are siblings in the structure.
- Each major section of content should normally begin with a heading.
- Keep heading text concise and descriptive.
- Heading text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** if it qualifies as large text.

### Code
```html
<h1>Web Development Guide</h1>

<h2>HTML Fundamentals</h2>
<p>Content about HTML...</p>

<p>Content about semantic elements...</p>
```

# D2. Paragraphs and Body Text

### Question  
> _What aspect of body text are you working on?_  

### Options  
- Color and contrast
- Spacing and line height
- Alignment and line length
- General text content

---

## D2-1. Text Color and Contrast

### Checklist
- Normal text under **18pt regular** and under **14pt bold** must have at least **4.5:1 contrast** against its background.
- Large text at **18pt and above**, or **14pt bold and above**, must have at least **3:1 contrast** against its background.
- Do not rely on color alone to convey meaning.
- Placeholder text should also meet applicable contrast requirements.
- Disabled text is exempt from contrast requirements, but avoid using low contrast for important information.
- Link text should have at least **4.5:1 contrast** against the background.
- If color alone distinguishes link text from surrounding non-link text, the link color needs at least **3:1 contrast** against the surrounding text.
- Text states such as hover, focus, active, and visited must remain readable.
- Consider users who override colors with browser or system settings.

### Code
```html
<p style="color: #333333; background-color: #ffffff;">  
This text has strong contrast.  
</p>  
  
<p style="color: #999999; background-color: #ffffff;">  
This text is too low contrast for normal body text.  
</p>
```
### Learn More
- Minimum contrast requirements: [WCAG 1.4.3 - Contrast (Minimum)](https://www.w3.org/WAI/WCAG22/quickref/#contrast-minimum)
- Use of color: [WCAG 1.4.1 - Use of Color](https://www.w3.org/WAI/WCAG22/quickref/#use-of-color)
- Contrast checking tool: [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- Understanding contrast: [W3C Understanding Contrast](https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html)

---

## D2-2. Text Spacing

### Checklist
- Support at least **1.5 times** font size for line height.
- Support at least **2 times** font size for spacing after paragraphs.
- Support at least **0.12 times** font size for letter spacing.
- Support at least **0.16 times** font size for word spacing.
- No content or functionality should be lost when users apply these text spacing values.
- Do not use fixed heights on containers that hold text.
- Text containers should expand when spacing changes.
- Test the page with user-overridden text spacing.

### Code
```css
body {
  line-height: 1.5;
}

p {
  margin-bottom: 2em;
  letter-spacing: 0.12em;
  word-spacing: 0.16em;
}

.text-box {
  min-height: 100px;
}
```

### Learn More
- Text spacing requirements: [WCAG 1.4.12 - Text Spacing](https://www.w3.org/WAI/WCAG22/quickref/#text-spacing)
- Understanding text spacing: [W3C Understanding Text Spacing](https://www.w3.org/WAI/WCAG22/Understanding/text-spacing.html)
- Text spacing testing bookmarklet: [Text Spacing Bookmarklet](https://www.html5accessibility.com/tests/tsbookmarklet.html)

---

## D2-3. Text Alignment and Line Length

### Checklist
- Use left alignment for left-to-right languages unless there is a clear reason not to.
- Use right alignment for right-to-left languages unless there is a clear reason not to.
- Avoid justified text because it can create uneven spacing between words.
- Keep line length to roughly **80 characters or fewer** where possible for long reading text.
- Use relative units such as `rem`, `em`, `%`, or `ch` rather than fixed widths that block reflow.
- Text should reflow without horizontal scrolling at **320 CSS pixels width** and at **400% zoom**, except where an allowed exception applies.
- Content should remain readable at **200% zoom**.
- Do not use `vw` alone as the only font-size unit.

### Code
```css
p {
  text-align: left;
  max-width: 65ch;
}

.content {
  max-width: 70rem;
  width: 100%;
}
```

### Learn More
- Reflow requirements: [WCAG 1.4.10 - Reflow](https://www.w3.org/WAI/WCAG22/quickref/#reflow)
- Resize text: [WCAG 1.4.4 - Resize Text](https://www.w3.org/WAI/WCAG22/quickref/#resize-text)
- Understanding reflow: [W3C Understanding Reflow](https://www.w3.org/WAI/WCAG22/Understanding/reflow.html)

## D2-4. General Text Content

### Checklist
- Use semantic HTML elements such as `<p>`, `<strong>`, and `<em>` where appropriate.
- Do not use `<br>` tags to create paragraph spacing.
- Do not use empty paragraphs for visual spacing.
- Ensure text can be selected and copied.
- Do not use images of text when real text can be used.
- Keep text content readable and well-structured.
- Use real paragraphs for body text.
- Body text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** if it qualifies as large text.

### Code
```html
<p>First paragraph of content with meaningful text.</p>
<p>Second paragraph continues the topic.</p>

<h1>Welcome to our site</h1>
```

- Images of text: [WCAG 1.4.5 - Images of Text](https://www.w3.org/WAI/WCAG22/quickref/#images-of-text)

# D3. Lists

### Question  
> _What type of list?_  

### Options  
- Unordered list (items with no particular order)
- Ordered list (items in a specific sequence)
- Description list (terms and definitions)
- Nested lists (lists within lists)

---

## D3-1. Unordered Lists

### Checklist
- Use `<ul>` and `<li>` for lists where item order is not meaningful.
- Do not create fake lists with `<div>` or `<p>` elements and visual dashes.
- Screen readers announce list semantics and number of items when proper markup is used.
- Each list item should be an `<li>`.
- Use unordered lists for groups such as features, options, or related links.

### Code
```html
<ul>
  <li>Feature one</li>
  <li>Feature two</li>
  <li>Feature three</li>
</ul>
```

- List elements: [MDN Unordered List](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/ul)

- Content structure tutorial: [W3C Content Structure](https://www.w3.org/WAI/tutorials/page-structure/content/)

---

## D3-2. Ordered Lists

### Checklist
- Use `<ol>` and `<li>` for sequential items.
- Screen readers announce position information such as item numbers when proper markup is used.
- Use ordered lists when sequence matters, such as steps, rankings, or instructions.
- Use the `start` attribute if the list does not begin at 1.
- Use the `reversed` attribute for countdown lists if needed.
- Do not use ordered lists when item order is not meaningful.

### Code
```html
<ol>
  <li>Open the settings page</li>
  <li>Click account preferences</li>
  <li>Select notification settings</li>
  <li>Save your changes</li>
</ol>

<ol start="5">
  <li>Fifth item</li>
  <li>Sixth item</li>
</ol>
```

- Content structure tutorial: [W3C Content Structure](https://www.w3.org/WAI/tutorials/page-structure/content/)

---

## D3-3. Description Lists

### Checklist
- Use `<dl>`, `<dt>`, and `<dd>` elements.
- Each term should be followed by one or more descriptions.
- Use description lists for key-value pairs, glossaries, metadata, and FAQs where appropriate.
- Do not use description lists for general page layout.
- Multiple `<dd>` elements can follow a single `<dt>`.

### Code
```html
<dl>
  <dt>Accessibility</dt>
  <dd>The practice of making websites usable by all people.</dd>

  <dt>WCAG</dt>
  <dd>Web Content Accessibility Guidelines.</dd>
```

### Checklist
- Place nested lists inside the parent `<li>`, not between list items.
- Keep nesting order correct so the nested list belongs to the correct parent item.
- Screen readers announce nesting level when semantics are correct.
- Avoid nesting too deeply; around **2 to 3 levels** is usually a practical readability limit.
- Each nested list should maintain proper semantic structure.
- Works for both ordered and unordered lists.

# D4. Tables

### Question  
> _What type of table?_  

### Options  
- Simple data table (one row of headers)
- Complex data table (multiple header rows or spanning cells)
- Layout table (not recommended)

---

## D4-1. Simple Data Tables

### Checklist
- Use `<table>`, `<thead>`, `<tbody>`, `<th>`, and `<td>` for data tables.
- Provide a table caption with `<caption>` when it helps identify the table.
- Column headers should use `<th scope="col">`.
- Row headers should use `<th scope="row">` when applicable.
- Do not use tables for visual layout.
- Ensure the table remains usable at smaller widths through responsive design, reflow-safe patterns, or another accessible presentation.
- Data cells should be clearly associated with their headers.
- Keep simple tables as simple as possible.
- Avoid empty header cells unless they are truly necessary.
- Table text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** if it qualifies as large text.

### Code
```html
<table>
  <caption>Employee Directory</caption>
  <thead>
    <tr>
      <th scope="col">Name</th>
      <th scope="col">Department</th>
      <th scope="col">Extension</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">Jane Smith</th>
      <td>Engineering</td>
      <td>4521</td>
    </tr>
    <tr>
      <th scope="row">John Doe</th>
      <td>Marketing</td>
      <td>3344</td>
    </tr>
  </tbody>
</table>
```

### Learn More
- Tables tutorial: [W3C Tables Tutorial](https://www.w3.org/WAI/tutorials/tables/)
- Simple tables: [W3C Tables with One Header](https://www.w3.org/WAI/tutorials/tables/one-header/)
- Info and relationships: [WCAG 1.3.1 - Info and Relationships](https://www.w3.org/WAI/WCAG22/quickref/#info-and-relationships)
- Table element reference: [MDN Table Element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/table)

---

## D4-2. Complex Data Tables

### Checklist
- Use header associations that correctly match the table structure.
- Each header that is referenced with `headers` must have a unique `id`.
- Each data cell using `headers` should reference all relevant header ids.
- Provide a table caption or summary text that explains the table structure when needed.
- Consider simplifying complex tables into smaller simpler tables where possible.
- Avoid merged cells where possible because they increase complexity.
- If merged cells are used, make sure associations remain correct.
- Test the table with a screen reader to verify reading order and header associations.
- Consider providing a text summary for very complex tables.
- Table text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** if it qualifies as large text.

### Code
```html
<table>
  <caption>Sales by Region and Quarter</caption>
  <thead>
    <tr>
      <td></td>
      <th id="q1" scope="col">Q1</th>
      <th id="q2" scope="col">Q2</th>
      <th id="q3" scope="col">Q3</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th id="north" scope="row">North</th>
      <td headers="north q1">$50k</td>
      <td headers="north q2">$65k</td>
      <td headers="north q3">$72k</td>
    </tr>
    <tr>
      <th id="south" scope="row">South</th>
      <td headers="south q1">$40k</td>
      <td headers="south q2">$55k</td>
      <td headers="south q3">$68k</td>
    </tr>
  </tbody>
</table>
```

- Tables with irregular headers: [W3C Irregular Headers](https://www.w3.org/WAI/tutorials/tables/irregular/)
- Info and relationships: [WCAG 1.3.1 - Info and Relationships](https://www.w3.org/WAI/WCAG22/quickref/#info-and-relationships)
- Using headers attribute: [W3C Using headers Attribute](https://www.w3.org/WAI/tutorials/tables/multi-level/)

## D4-3. Layout Tables

### Checklist
- Avoid using tables for layout.
- Use CSS layout techniques such as flexbox or grid instead.
- If a layout table must be used for legacy reasons, use `role="presentation"` on the table.
- Layout tables must not use `<caption>`, `<summary>`, `<th>`, `scope`, or `headers`.
- Refactor layout tables to CSS as soon as possible.
- Content inside layout tables must still preserve a logical reading order.

### Code
```html
<table role="presentation">  
<tr>  
<td>Left column content</td>  
<td>Right column content</td>  
</tr>  
</table>  
  
<div class="layout">  
<div class="left">Left column content</div>  
<div class="right">Right column content</div>  
</div>
```

### Code
```css
.layout {  
  display: flex;  
  gap: 1rem;  
}
```

# D5. Quotes

> _What type of quote?_  

### Options  
- Short inline quote
- Long block quote

---

## D5-1. Inline Quotes

### Checklist
- Use the `<q>` element for short inline quotes.
- Browsers typically add quotation marks automatically for `<q>`.
- Add the `cite` attribute if a source URL is available.
- Do not manually add quotation marks around a `<q>` element unless you have a special reason.
- Ensure quoted text remains readable and meets text contrast requirements.

### Code
```html
<p>
  As the guideline states,
  <q cite="https://www.w3.org/WAI/fundamentals/accessibility-intro/">
    The web is fundamentally designed to work for all people
  </q>.
</p>
```

### Learn More
- Quote element reference: [MDN q Element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/q)
- Info and relationships: [WCAG 1.3.1 - Info and Relationships](https://www.w3.org/WAI/WCAG22/quickref/#info-and-relationships)

---

## D5-2. Block Quotes

### Checklist
- Use `<blockquote>` for longer quotations.
- Add the `cite` attribute for the source URL when available.
- Include visible attribution using `<cite>` or another suitable element when needed.
- Do not use `<blockquote>` only for visual indentation.
- Ensure quote text meets contrast requirements.
- Keep the original structure of the quoted content where appropriate, such as multiple paragraphs.

### Code

```html
<blockquote cite="https://www.w3.org/WAI/fundamentals/">
```

# D6. Text Emphasis and Semantic Styling

### Question  
> _What type of text emphasis?_  

### Options  
- Important or strong emphasis
- Stress emphasis (italic)
- Deleted or inserted text
- Highlighted or marked text
- Code or technical text

---

## D6-1. Strong Emphasis

### Checklist
- Use `<strong>` for text with strong importance.
- Do not use `<strong>` only for visual bold styling.
- Screen readers may announce strong text with changed tone or emphasis depending on the technology.
- Use CSS `font-weight` for purely visual bold text.
- Strong emphasis may be nested for increased importance when it is meaningful.

### Code
```html
<p><strong>Warning:</strong> Do not delete this file.</p>
<p>The deadline is <strong>Friday, March 15</strong>.</p>

<p><span style="font-weight: bold;">Company Name</span></p>
```

---

## D6-2. Stress Emphasis (Italic)

### Checklist
- Use `<em>` for text whose stress emphasis changes meaning.
- Do not use `<em>` only for visual italic styling.
- Screen readers may announce emphasized text differently depending on the technology.
- Use CSS `font-style` for purely visual italic styling.
- Use `<i>` when you need alternate voice or mood without stress emphasis, such as technical terms or foreign words when that is the right semantic fit.

### Code
```html
<p>You need to <em>submit</em> the form before Friday.</p>
<p>The term <i>accessibility</i> comes from Latin.</p>

<p><span style="font-style: italic;">Photo by Jane Smith</span></p>
```

## D6-3. Deleted and Inserted Text

### Checklist
- Use `<del>` for removed content.
- Use `<ins>` for added content.

- Consider adding hidden explanatory text if the change is important to understand.
- Do not rely only on strikethrough or underline styling to convey meaning.
- Provide context for why content changed when needed.


- Consider adding hidden explanatory text if the highlight is important to understand.
- Ensure highlighted text still meets contrast requirements.
- Do not rely on highlight color alone to convey meaning.

# D7. Font Size and Zoom

## D7. Font Size and Zoom

### Checklist
- Use relative units such as `rem`, `em`, `%`, `ch`, or `clamp()` rather than fixed pixel font sizes where possible.
- The base text size should effectively support at least **16px** equivalent for body text.
- Allow browser zoom up to **200%** without loss of content or functionality.
- Do not disable zoom with viewport meta settings like `maximum-scale=1` or `user-scalable=no`.
- Avoid horizontal scrolling at **320 CSS pixels width**, which corresponds to **400% zoom** on a 1280px-wide viewport, except where an allowed exception applies.
- Content should reflow when zoomed so text is not cut off.
- Do not use `vw` alone as the sole font-size unit.
- Test the page at **200% browser zoom**.
- Ensure interactive elements remain usable when zoomed.

### Code
```html
<meta name="viewport" content="width=device-width, initial-scale=1">
```
### CSS
```css
body {
  font-size: 1rem;
}

h1 {
  font-size: 2rem;
}

h2 {
  font-size: 1.5rem;
}

p {
  font-size: 1rem;
}

.hero-title {
  font-size: clamp(1.5rem, 3vw, 3rem);
}
```

### Learn More
- Resize text: [WCAG 1.4.4 - Resize Text](https://www.w3.org/WAI/WCAG22/quickref/#resize-text)
- Reflow: [WCAG 1.4.10 - Reflow](https://www.w3.org/WAI/WCAG22/quickref/#reflow)
- Understanding reflow: [W3C Understanding Reflow](https://www.w3.org/WAI/WCAG22/Understanding/reflow.html)
- CSS relative units: [MDN CSS Values and Units](https://developer.mozilla.org/en-US/docs/Learn/CSS/Building_blocks/Values_and_units)

## D8-1. Page Language

### Checklist
- Set the `lang` attribute on the `<html>` element.
- Use the correct language code, such as `en` for English or `de` for German.
- The page language affects screen reader pronunciation and other assistive technology behavior.
- The selected language should match the main language of the page.
- Use valid language codes based on recognized standards.
- Regional variants can be specified when needed, such as `en-US` or `en-GB`.

```html
<html lang="en">
```

## D8-2. Language Changes Within a Page

### Checklist
- Use the `lang` attribute on elements that contain text in a different language.
- Apply the language attribute to the smallest suitable element containing the foreign-language text.

- It can apply to individual words, phrases, or larger sections.
- Common borrowed words do not always need separate language markup if they are fully naturalized in the page language.

# E1. Buttons

### Question  
> _What does the button do?_  

### Options  
- Submits a form
- Triggers an action on the page (save, delete, play, etc.)
- Opens something (modal, menu, popover)
- It looks like a button but takes you to another page

---

## E1-1. Button That Navigates to Another Page

If your “button” takes the user to a different page or URL, it should be a link (`<a>`), not a `<button>`.

### How to decide
- Takes the user somewhere new → use an `<a>` element
- Does something on the current page → use a `<button>`
- Submits a form → use `<button type="submit">`

### Question  
> _What applies here?_  

### Options  
- My button navigates to a URL

### Learn More
- Links and navigation: [WCAG 2.4.4 - Link Purpose (In Context)](https://www.w3.org/WAI/WCAG22/quickref/#link-purpose-in-context)

---

## E1-2. Form Submit Buttons

### Checklist
- Use a real `<button>` element with `type="submit"`.
- Button text should describe the action, for example “Create account” rather than only “Submit” when more specific wording helps.
- The button should be inside the form it submits, or otherwise programmatically associated with the form.
- Submit buttons must be keyboard accessible.
- Users should see a visible focus indicator when navigating by keyboard.
- Do not disable the submit button in a way that prevents users from discovering what is missing; prefer validation help and error messaging.
- If the form submission takes time, communicate the loading state.
- Button text must meet at least **4.5:1 contrast** against the button background for normal-sized text, or **3:1** for large text.
- Visual boundaries and focus indicators relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- Pointer targets should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<form action="/register" method="post">
  <button type="submit">Create Account</button>
</form>

<form action="/search" method="get">
  <label for="search">Search</label>
  <input type="search" id="search" name="q">
  <button type="submit">Search</button>
</form>
```

- Button element reference: [MDN Button Element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/button)
- Name role value: [WCAG 4.1.2 - Name, Role, Value](https://www.w3.org/WAI/WCAG22/quickref/#name-role-value)
- Keyboard accessible: [WCAG 2.1.1 - Keyboard](https://www.w3.org/WAI/WCAG22/quickref/#keyboard)

## E1-3. Action Buttons

### Question  
> _Does the button have visible text?_  

### Options  
- Yes, text label is visible
- No, icon only

---

### E1-3-1. Text Action Buttons

#### Checklist
- Use a real `<button type="button">` for in-page actions.
- The visible text should clearly describe the action.
- Buttons must be keyboard accessible with Enter and Space.
- Provide a visible focus indicator.
- If the button includes both an icon and text, make the icon decorative with `aria-hidden="true"`.
- Do not use `<div>` or `<span>` as buttons.
- If the button changes state, communicate that change.
- Provide sufficient contrast for all states, including default, hover, focus, active, and disabled.
- Target size should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
- Button text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

#### Code
```html
<button type="button">Save Changes</button>

<button type="button">
  <svg aria-hidden="true" viewBox="0 0 24 24" focusable="false">
    <use href="#delete-icon"></use>
  </svg>
  Delete Item
</button>

<button type="button">Remove from Cart</button>
```

- Focus visible: [WCAG 2.4.7 - Focus Visible](https://www.w3.org/WAI/WCAG22/quickref/#focus-visible)

---

### E1-3-2. Icon-Only Buttons

#### Checklist
- Use a real `<button type="button">`.
- Provide an accessible name with `aria-label` or with visually hidden text inside the button.
- Hide the icon itself from assistive technologies with `aria-hidden="true"`.
- The accessible name should describe the action, not the icon appearance.
- A visible tooltip may help, but it must not be the only accessible name.
- Buttons must be keyboard accessible.
- Provide a visible focus indicator.
- Target size should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
- The icon meaning should be widely understood, or be supported by additional text.

#### Code
```html
<button type="button" aria-label="Close dialog">
  <svg aria-hidden="true" viewBox="0 0 24 24" focusable="false">
    <use href="#close-icon"></use>
  </svg>
</button>

<button type="button">

  </svg>
  <span class="visually-hidden">Print this page</span>
</button>

<button type="button" aria-label="Edit profile">

    <use href="#edit-icon"></use>
  </svg>
</button>
```

## E1-4. Buttons That Open Things

### E1-4-5. Buttons That Open Things (General)

#### Checklist
- Use a real `<button type="button">`.
- Use `aria-expanded="false"` when collapsed and update it to `true` when opened, if the control reveals and hides content.
- Use `aria-controls` to point to the id of the controlled content when that relationship is useful.
- The button text or label should describe what opens.
- The button must be keyboard accessible with Enter and Space.
- Provide a visible focus indicator.

```html
<button
  type="button"

  aria-controls="content-panel">

</button>

  Content that appears when the button is activated.
</div>
```

# E2. Modal Dialogs

### Question  
> _What type of dialog?_  

### Options  
- Alert or confirmation dialog (user must respond)
- General dialog/modal (content, forms, settings)

---

## E2-1. Alert/Confirmation Dialogs

### Checklist
- Use `role="alertdialog"` on the dialog container.
- Provide a clear title using a heading element.
- Associate the title with the dialog using `aria-labelledby`.
- Associate descriptive text with the dialog using `aria-describedby` when useful.
- Move focus into the dialog when it opens, usually to the first focusable element or the dialog itself if appropriate.
- Trap keyboard focus inside the dialog while it is open.
- Escape should close the dialog when that is appropriate.
- Close, Cancel, and Confirm buttons must be clearly labeled.
- When the dialog closes, return focus to the control that opened it.
- Background content must not be interactive while the dialog is open.
- The action buttons must clearly describe the choice, for example “Delete item” and “Cancel,” not only “Yes” and “No.”
- Dialog text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<button type="button" id="delete-trigger">Delete Item</button>

<div
  role="alertdialog"
  aria-modal="true"
  aria-labelledby="alert-title"
  aria-describedby="alert-desc"
  hidden>
  <h2 id="alert-title">Confirm Deletion</h2>
  <p id="alert-desc">Are you sure you want to delete this item? This action cannot be undone.</p>

  <button type="button">Cancel</button>
  <button type="button">Delete Item</button>
</div>
```

### Learn More
- Alert dialog pattern: [ARIA Alert Dialog Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/alertdialog/)
- Focus management: [WCAG 2.4.3 - Focus Order](https://www.w3.org/WAI/WCAG22/quickref/#focus-order)
- Error prevention: [WCAG 3.3.4 - Error Prevention](https://www.w3.org/WAI/WCAG22/quickref/#error-prevention-legal-financial-data)

---

## E2-2. General Modal Dialogs

### Checklist
- Use `role="dialog"` on the dialog container.
- Set `aria-modal="true"` on the modal dialog.
- Provide a visible title using a heading element.
- Link the title to the dialog using `aria-labelledby`.
- Link descriptive text using `aria-describedby` when there is additional context.
- Move focus into the dialog when it opens.
- Trap focus inside the dialog while it is open.
- Escape should close the dialog.
- Provide a visible close button.
- When the dialog closes, return focus to the control that opened it.
- Background content must not be interactive while the dialog is open.
- The page behind the dialog should not scroll while the modal is open.
- All focusable elements inside the dialog must follow normal keyboard accessibility rules.
- Dialog text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- Focus indicators and visual boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<button
  type="button"
  aria-expanded="false"
  aria-controls="settings-dialog">
  Edit Settings
</button>

<div
  id="settings-dialog"
  role="dialog"
  aria-modal="true"
  aria-labelledby="dialog-title"
  hidden>
  <h2 id="dialog-title">Edit Profile Settings</h2>

  <form>
    <label for="display-name">Display Name</label>
    <input type="text" id="display-name" name="display-name">

    <label for="bio">Bio</label>
    <textarea id="bio" name="bio"></textarea>

    <button type="button">Cancel</button>
    <button type="submit">Save Changes</button>
  </form>

  <button type="button" aria-label="Close dialog">×</button>
</div>
```

# E3. Tooltips

### Question  
> _Is the tooltip content essential to complete the task?_  

### Options  
- Yes, user needs this information
- No, it provides supplementary context

---

## E3-1. Essential Information in Tooltips

Do not put essential information only in a tooltip. If users need the information to complete a task, it should be visible by default.

### Better alternatives
- Show helper text directly below the input or element
- Use `aria-describedby` to connect help text to the control
- Display instructions in a visible paragraph

### Code
```html
<label for="username">Username</label>
<input type="text" id="username" aria-describedby="username-help">
<p id="username-help">Username must be 3–20 characters and use letters and numbers only.</p>
```

### Learn More
- Labels and instructions: [WCAG 3.3.2 - Labels or Instructions](https://www.w3.org/WAI/WCAG22/quickref/#labels-or-instructions)
- Content on hover or focus: [WCAG 1.4.13 - Content on Hover or Focus](https://www.w3.org/WAI/WCAG22/quickref/#content-on-hover-or-focus)

---

## E3-2. Tooltips

### Checklist
- Tooltips should appear on both mouse hover and keyboard focus.
- Use `aria-describedby` to associate the tooltip with its trigger.
- The tooltip itself should use `role="tooltip"`.
- The tooltip should stay visible while the user hovers over the trigger or the tooltip itself.
- Users must be able to dismiss the tooltip with Escape without moving focus.
- The tooltip must not obscure the trigger element in a way that blocks use.
- Keep tooltip text concise.
- The tooltip should not disappear until the user moves away or dismisses it.
- Provide a mobile-friendly alternative because hover-only tooltips do not work well on touch devices.
- Tooltip text must meet at least **4.5:1 contrast** against the tooltip background for normal text, or **3:1** for large text.
- The tooltip boundary, pointer, and focus-related visuals relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<button
  type="button"
  aria-describedby="save-tooltip"
  aria-label="Save">
  <svg aria-hidden="true" viewBox="0 0 24 24" focusable="false">
    <use href="#save-icon"></use>
  </svg>
  <span class="visually-hidden">Save</span>
</button>

<div role="tooltip" id="save-tooltip" hidden>
  Save your current progress
</div>
```

- Tooltip pattern: [ARIA Tooltip Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/tooltip/)
- Content on hover or focus: [WCAG 1.4.13 - Content on Hover or Focus](https://www.w3.org/WAI/WCAG22/quickref/#content-on-hover-or-focus)

# E4. Tabs

## E4. Tab Interfaces

### Checklist
- The tab list container should use `role="tablist"`.
- Give the tab list an accessible name with `aria-label` or `aria-labelledby` when needed.
- Each tab should use `role="tab"`.
- Each tab panel should use `role="tabpanel"`.
- The active tab should have `aria-selected="true"`.
- Inactive tabs should have `aria-selected="false"`.
- Each tab should point to its panel id with `aria-controls`.
- Each panel should point back to its tab with `aria-labelledby`.
- Only the active tab should be in the normal Tab sequence.
- Inactive tabs should be reachable with arrow keys.
- Keyboard users should be able to move from the active tab into the active panel.
- Left and right arrow keys should move between tabs in horizontal tablists. Up and down may be used in vertical tablists.
- Home should move to the first tab and End to the last tab when that pattern is supported.
- Provide a visible focus indicator on the active and focused tab.
- Tab text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- The visual indicator of the selected tab, focus ring, and control boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<div role="tablist" aria-label="Profile settings">
  <button role="tab" id="tab-1" aria-selected="true" aria-controls="panel-1">Profile</button>
  <button role="tab" id="tab-2" aria-selected="false" aria-controls="panel-2">Security</button>
  <button role="tab" id="tab-3" aria-selected="false" aria-controls="panel-3">Notifications</button>
</div>

<div role="tabpanel" id="panel-1" aria-labelledby="tab-1">
  <p>Profile settings content.</p>
</div>
```

# E5. Accordions

### Checklist
- Each accordion header should contain a real button element.
- Buttons should be wrapped in an appropriate heading level such as `<h2>` or `<h3>`.
- Each button should use `aria-expanded="true"` or `aria-expanded="false"`.
- Each button should point to its panel id with `aria-controls`.

# E6. Carousels/Sliders

## E6. All Carousels (General)

### Checklist
- The carousel container should have an accessible label that describes its purpose.
- The carousel container should have `aria-roledescription="carousel"` when that helps clarify the widget.
- Each slide should be wrapped in a container with `role="group"`.
- Each slide should have `aria-roledescription="slide"`.
- Each slide should have an accessible label indicating its position, such as “1 of 5”.
- Previous and Next buttons must have clear labels.
- The current slide and total number of slides should be communicated.
- All controls must be keyboard accessible.
- Navigation controls should have an adequate target size of at least **24 × 24 CSS pixels**, unless a WCAG exception applies.
- Provide a visible focus indicator on all controls.
- Slides that are not visible should be hidden from assistive technologies if they are not meant to be read.

# E7. Toggle Switches

### Question  
> _Is this a simple on/off toggle or a setting with more than two states?_  

### Options  
- Simple on/off toggle
- Setting with multiple states

---

## E7-1. Toggle Switch

### Checklist
- Use a real button with `role="switch"` or a real checkbox when that pattern fits better.
- Provide a visible label that describes what the control changes.
- Use `aria-checked="true"` or `aria-checked="false"` to communicate state for switch widgets.
- Space must toggle the switch.
- The current state should be announced to screen reader users when toggled.
- The visual design must clearly show on and off states.
- Do not rely on color alone to show state; use position, text, or icons as well.
- The label should not change completely when the state changes; users need a stable name.
- Provide a visible focus indicator.
- Consider whether a simple checkbox would work better.
- Label text must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- The state indicator, focus ring, and control boundary relied on for understanding should have at least **3:1 contrast** against adjacent colors.
- The target size should be at least **24 × 24 CSS pixels**, unless a WCAG exception applies.

### Code
```html
<label id="notif-label">Email Notifications</label>
<button
  type="button"
  role="switch"
  aria-checked="false"
  aria-labelledby="notif-label">
  <span aria-hidden="true">Off</span>
</button>

<label for="dark-mode">Dark Mode</label>
<input
  type="checkbox"
  role="switch"
  id="dark-mode"
  name="dark-mode">
```

- Switch pattern: [ARIA Switch Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/switch/)

- Name role value: [WCAG 4.1.2 - Name, Role, Value](https://www.w3.org/WAI/WCAG22/quickref/#name-role-value)
- Use of color: [WCAG 1.4.1 - Use of Color](https://www.w3.org/WAI/WCAG22/quickref/#use-of-color)

## E7-2. More Than Two States

A toggle switch is not the right pattern for this.

Toggle switches are designed for exactly two states: on and off. When you have more than two options, a toggle creates confusion because users cannot tell what the other possible values are.

### Why this matters
- A toggle only communicates two states to screen readers and other assistive technologies.
- Users cannot discover all available options from a two-state switch.
- It breaks user expectations of what a toggle does.

### Question  
> _What should be used instead?_  

### Options  
- Radio buttons
- Select dropdowns

# E8. Autocomplete/Typeahead

## E8. Autocomplete Fields

### Checklist
- The input should use `role="combobox"` when implementing a custom autocomplete.
- The input should use `aria-autocomplete="list"` or `aria-autocomplete="both"` as appropriate.
- The input should have `aria-expanded="false"` when suggestions are hidden and `aria-expanded="true"` when suggestions are shown.
- The input should point to the suggestion list with `aria-controls`.
- The suggestion list should use `role="listbox"`.
- Each suggestion should use `role="option"`.
- When navigating suggestions with arrow keys, use `aria-activedescendant` on the input to indicate the highlighted option in custom implementations.
- Enter should select the highlighted suggestion.
- Escape should close the suggestion list.
- Arrow keys should navigate through suggestions.
- Typing should filter the list.
- Announce the number of results when the list updates, ideally through a polite live region.
- All suggestions must be keyboard accessible.
- The currently highlighted suggestion should have a clear visible focus or highlight state.
- The input must have a visible label.
- Text labels and suggestions must meet at least **4.5:1 contrast** against the background for normal text, or **3:1** for large text.
- The focus indicator, highlighted option, and list boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<label for="country-search">Country</label>
<input
  type="text"
  id="country-search"
  role="combobox"
  aria-autocomplete="list"
  aria-expanded="false"
  aria-controls="country-list"
  aria-activedescendant=""
  autocomplete="off">

<ul id="country-list" role="listbox" hidden>
  <li role="option" id="opt-us">United States</li>
  <li role="option" id="opt-uk">United Kingdom</li>
  <li role="option" id="opt-ca">Canada</li>
</ul>

<div aria-live="polite" class="visually-hidden">
  3 results available
</div>
```

- Combobox pattern: [ARIA Combobox Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/combobox/)
- Listbox pattern: [ARIA Listbox Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/listbox/)
- Keyboard: [WCAG 2.1.1 - Keyboard](https://www.w3.org/WAI/WCAG22/quickref/#keyboard)

# F1. Landmarks and Regions

## F1-1. Header/Banner Landmark

### Checklist
- Use the `<header>` element as a direct child of `<body>` for the page-level banner.
- Only one page-level banner landmark should exist per page.
- The banner should contain site-wide content such as logo, main navigation, or search.
- Do not assume every `<header>` is a banner; headers inside articles or sections do not create page-level banner landmarks.
- If multiple `<header>` elements exist, only the top-level page header should act as the banner.
- Do not place the page banner inside `<main>` or `<aside>`.
- Landmark labels and text inside the banner must meet text contrast requirements: **4.5:1** for normal text, **3:1** for large text.

### Code
```html
<body>
  <header>
    <a href="/">
      <img src="logo.png" alt="Company Name - Home">
    </a>

    <nav aria-label="Main navigation">
      <ul>
        <li><a href="/">Home</a></li>
        <li><a href="/about">About</a></li>
        <li><a href="/contact">Contact</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <!-- page content -->
  </main>
</body>
```

### Learn More
- Banner landmark: [W3C Page Regions - Header](https://www.w3.org/WAI/tutorials/page-structure/regions/)
- Header element: [MDN Header Element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/header)
- Landmark regions: [ARIA Landmark Roles](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Roles/landmark_role)

## F1-2. Main Content Landmark

### Checklist
- Use exactly one `<main>` element per page.
- `<main>` should contain the primary content of the page.
- Skip links should point to the main content region.
- Do not place `<main>` inside `<header>`, `<footer>`, `<aside>`, or `<nav>`.
- Main content should not include repeated site-wide content such as global navigation, footer, or repeated sidebars.
- `<main>` should usually be a direct child of `<body>` or of a neutral wrapper that does not change meaning.
- Do not use `role="main"` on a generic element when you can use the native `<main>` element instead.

### Code
```html
<body>
  <a href="#main" class="skip-link">Skip to main content</a>

  <header>
    <!-- site header -->
  </header>

  <main id="main">
    <h1>Page Title</h1>
    <p>Primary page content goes here...</p>
  </main>

  <footer>
    <!-- site footer -->
  </footer>
</body>
```

### Learn More
- Main landmark: [W3C Page Regions - Main](https://www.w3.org/WAI/tutorials/page-structure/regions/)
- Main element: [MDN Main Element](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/main)

## F1-3. Footer/Contentinfo Landmark

### Checklist
- Use the `<footer>` element as a direct child of `<body>` for the page-level contentinfo landmark.
- Only one page-level contentinfo landmark should exist per page.
- The page footer should contain site-wide information such as copyright, legal links, contact, or footer navigation.
- A `<footer>` inside an article or section does not create the page-level contentinfo landmark.
- If several footers exist, only the top-level page footer should be the contentinfo landmark.
- Do not place the page footer inside `<main>`.
- Footer text and labels must meet text contrast requirements: **4.5:1** for normal text, **3:1** for large text.

### Code
```html
<body>
  <header>
    <!-- site header -->
  </header>

  <main>
    <!-- page content -->
  </main>

  <footer>
    <nav aria-label="Footer navigation">
      <ul>
        <li><a href="/privacy">Privacy Policy</a></li>
        <li><a href="/terms">Terms of Service</a></li>
      </ul>
    </nav>
    <p>Copyright 2024 Company Name. All rights reserved.</p>
  </footer>
</body>
```

## F1-4. Aside/Complementary Landmark

### Checklist
- Use `<aside>` for content related to, but separate from, the main content.
- The content should make sense on its own while still being related to the surrounding page or section.
- Give each complementary region a descriptive accessible name if there are multiple sidebars or asides.
- Common uses include related links, related articles, sidebar widgets, or glossary panels.
- An aside can appear inside or outside `<main>` depending on what it relates to.
- If it is inside `<main>`, it usually relates to nearby content. If it is outside `<main>`, it usually relates to the page as a whole.
- Text and labels inside the aside must meet text contrast requirements: **4.5:1** for normal text, **3:1** for large text.

### Code
```html
<main>
  <h1>Article Title</h1>
  <p>Main article content...</p>
</main>

<aside aria-label="Related articles">
  <h2>Related Articles</h2>
  <ul>
    <li><a href="/article-1">Related Article One</a></li>
    <li><a href="/article-2">Related Article Two</a></li>
  </ul>
</aside>

<!-- another example -->
<main>
  <h1>Guide to Accessibility</h1>
  <p>Content about accessibility...</p>

  <aside aria-label="Quick tips">
    <h2>Quick Tips</h2>
    <p>Always test your site with a screen reader.</p>
  </aside>
</main>
```

## F1-5. Form as Landmark

### Checklist
- Use the `<form>` element for forms.
- A form with an accessible name becomes a landmark that screen reader users can navigate to.
- Give meaningful forms an accessible name with `aria-label`, `aria-labelledby`, or a visible heading associated to the form.
- Only forms that represent important page sections, such as search, contact, or registration, should be treated as landmarks.
- Do not label every tiny inline form as a landmark.
- The heading inside the form can be used with `aria-labelledby`.

### Code
```html
<form aria-label="Contact us">
  <h2>Contact Us</h2>

  <label for="name">Name</label>
  <input type="text" id="name" name="name">

  <label for="message">Message</label>
  <textarea id="message" name="message"></textarea>

  <button type="submit">Send Message</button>
</form>
```

## F1-7. Section/Region Landmark

### Checklist
- Use `<section>` for distinct content sections.
- A section becomes a landmark only when it has an accessible name.
- Label the section with `aria-label` or `aria-labelledby` when it should be a region landmark.
- Use sections when content has a clear theme or purpose.
- Each section should usually have a heading.
- Do not use `<section>` as a generic wrapper when no thematic grouping exists.
- Do not create too many region landmarks because that adds noise for screen reader users.
- Only mark sections as regions when doing so helps navigation.

### Code
```html
<section aria-labelledby="features-heading">
  <h2 id="features-heading">Features</h2>
  <p>Our product includes...</p>
</section>

<section aria-labelledby="pricing-heading">
  <h2 id="pricing-heading">Pricing</h2>
  <p>We offer three plans...</p>
</section>
```

## F1-9. Choosing Landmarks

Every page should usually include these core landmarks at minimum:
- Header/banner
- Main content
- Footer/contentinfo

Optional landmarks depend on the page:
- Navigation for menus
- Aside/complementary for side content
- Form landmark for important named forms
- Search landmark for search functionality
- Region landmarks for distinct named content sections

### Why landmarks matter
- Screen reader users can jump directly between landmarks.
- Landmarks create a clear map of the page.
- They help users skip to the section they need.
- Assistive technologies announce landmarks as users navigate.

# F2. Page Title

## F2. Page Title

### Checklist
- Every page must have a unique `<title>` element in the document `<head>`.
- The title should describe the page content and purpose.
- The title should identify the page or view within the site or application.
- Common formats include “Page Name - Site Name” or “Page Name | Site Name”.
- Make the title specific to the page, not the same title reused everywhere.
- For single-page apps, update the title when the view changes.
- The title is often the first thing screen readers announce for a page.

- Avoid keyword stuffing or unnecessarily long titles.

## F3-1. Article Element

## F3-2. Figure and Figcaption

### Checklist
- Use `<figure>` for self-contained content referenced from the main text.
- Use `<figcaption>` for the caption, usually as the first or last child of the figure.
- Common uses include images, diagrams, code snippets, charts, and quotations.
- Figure semantics do not replace proper alt text on images.
- The figcaption should be visible to all users.
- If both image alt text and figcaption are present, they should complement rather than duplicate each other.
- The figure should be movable or removable without breaking the surrounding document flow.

## F3-3. Content Grouping

### Checklist
- Use `<div>` for generic grouping only when no semantic element fits.
- Use `<section>` when the content has a clear topic and usually a heading.
- Use `<article>` when the content is self-contained.
- Do not use `<section>` or `<article>` as generic wrappers.
- Every section should normally have a heading.
- Choose the element that best matches the content type.

# F4. Responsive Layout

### Question  
> _What responsive concern do you have?_  

### Options  
- Content reflow when zoomed
- Mobile layout and touch targets
- Hiding/showing content at different sizes
- Orientation

---

## F4-1. Content Reflow

### Checklist
- Content should reflow at **320 CSS pixels width**, which is equivalent to **400% zoom** on a 1280px-wide viewport.
- No horizontal scrolling should be required at 320px width for primary content, except where WCAG allows an exception.
- No vertical clipping at **256px height** should hide content.
- Do not use fixed widths that break at smaller sizes.
- Use relative units such as `%`, `rem`, `em`, `vw` carefully, or fluid layouts instead of rigid fixed widths.
- Ensure all content remains accessible when reflowed.
- No content should be cut off or hidden behind other content.
- Do not overlap text or controls.
- Test by zooming the browser to **400%**.
- Two-dimensional content such as data tables, maps, or some diagrams can be exceptions when necessary.

### Code
```css
.container {
  max-width: 80rem;
  width: 100%;
  padding: 0 1rem;
}

.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(16rem, 1fr));
  gap: 1rem;
}

h1 {
  font-size: clamp(1.75rem, 4vw, 3rem);
}
```

## F4-2. Mobile/Touch Accessibility

### Checklist
- Allow zooming; do not block pinch zoom with restrictive viewport settings.
- Interactive controls should meet a minimum target size of **24 × 24 CSS pixels**, unless a WCAG exception applies.
- Larger targets such as **44 × 44 CSS pixels** are often easier to use on touch devices.
- Provide enough spacing between adjacent targets to reduce accidental taps.
- Make links and buttons easy to activate on small screens.
- Ensure touch interactions also have keyboard and screen reader equivalents where needed.

## F4-3. Responsive Visibility

### Checklist
- Content hidden on mobile should not contain essential information unless that information is available in another accessible way.
- If content is hidden visually, it should also be hidden from screen readers when it is truly unavailable.
- Use `display: none` or the `hidden` attribute to fully hide content.
- Do not use `visibility: hidden` when the content still occupies layout space without good reason.
- If content is shown on mobile that was hidden on desktop, ensure it is accessible and understandable in both contexts.
- Mobile navigation such as hamburger menus must be fully accessible.
- Hamburger buttons should have clear labels.
- Buttons that open and close responsive menus should use `aria-expanded`.
- Reconsider whether hiding the content is actually the best solution.

## F4-4. Screen Orientation

### Checklist
- Do not lock content to portrait only or landscape only unless an allowed exception applies.
- Content should adapt to both orientations.
- All content and functionality should be available in both orientations.
- A few specialized activities may be exceptions if they naturally require one orientation.
- Use CSS media queries to adapt layout, not to restrict orientation.
- Test in both portrait and landscape.

# F5. Reading Order

## F5. Reading Order

### Checklist
- The visual order should match the DOM order.
- Screen readers read content in DOM order, not in visual order.
- Do not use CSS to rearrange content in a way that breaks reading order.
- Be careful with Flexbox `order` because it can change visual order without changing reading order.
- Be careful with CSS Grid placement for the same reason.
- Tab order should match visual order.
- Test by disabling CSS and checking whether the content still makes sense.
- Test with the Tab key to verify the order matches the visual layout.
- Right-to-left languages need `dir="rtl"` on the appropriate element.

### Code
```html
<header>Site header</header>
<nav>Navigation</nav>
<main>Main content</main>
<aside>Sidebar</aside>
<footer>Footer</footer>
```

```css
.container {
  display: flex;
}

/* Avoid changing visual order in a way that breaks reading order */
.sidebar {
  order: -1;
}
```

# F6. Focus Order

## F6. Focus Order

### Checklist
- Tabbing order should follow a logical sequence.
- Focus order should match the visual layout.
- Avoid positive `tabindex` values such as `1`, `2`, or higher.
- Use `tabindex="0"` only when a normally non-focusable element must become focusable for a valid reason.
- Use `tabindex="-1"` when an element needs programmatic focus but should not be in the normal Tab sequence.
- Interactive elements such as links, buttons, inputs, and controls should normally be tabbable by default.
- Modal dialogs must trap focus while open and restore focus when closed.
- When new content appears, manage focus appropriately.
- After deleting an item, move focus to a logical location.
- Focus should not jump unexpectedly.
- Focus should never get lost, such as moving to `<body>` with no clear location.
- Test the entire page with the Tab key.

### Code
```html
<!-- Good: natural tab order follows DOM -->
<nav>
  <a href="/">Home</a>
  <a href="/about">About</a>
  <a href="/contact">Contact</a>
</nav>

<!-- Bad: positive tabindex creates unnatural focus order -->
<a href="/contact" tabindex="3">Contact</a>
<a href="/" tabindex="1">Home</a>
<a href="/about" tabindex="2">About</a>

<!-- Good: tabindex="-1" for programmatic focus only -->
<h2 id="results" tabindex="-1">Search Results</h2>
<!-- JavaScript can move focus here after results update -->

<!-- Good: tabindex="0" only when a non-interactive container
     must become keyboard focusable -->
<div tabindex="0" role="region" aria-label="Scrollable code example">
  <pre><code>Very long code content...</code></pre>
</div>
```

- Focus order: [WCAG 2.4.3 - Focus Order](https://www.w3.org/WAI/WCAG22/quickref/#focus-order)

# F7. Scrollable Regions

## F7. Scrollable Regions

### Checklist
- Scrollable regions must be keyboard accessible.
- If content overflows and creates a scroll, make the container focusable when needed.
- Use `tabindex="0"` on a scrollable container if it has no naturally focusable children and users need to scroll it with the keyboard.
- Provide visible scroll indicators.
- Ensure the scrollbar or scroll indicator has enough contrast if users rely on it visually.
- Do not rely on horizontal scrolling for primary content unless an exception applies.
- For code blocks or data tables, horizontal scrolling can be acceptable.
- Test whether keyboard users can scroll the region.
- Give custom scrollable regions an accessible name with `role="region"` and `aria-label` when that helps users identify the area.
- Focus indicators and scroll container boundaries relied on for understanding should have at least **3:1 contrast** against adjacent colors.

### Code
```html
<div
  role="region"
  aria-label="Code example"
  tabindex="0"
  class="code-scroll">
  <code>/... very long line of code that extends beyond the container width .../</code>
</div>

<div
  role="region"
  aria-label="Employee directory table"
  tabindex="0"
  class="table-scroll">
  <table>
    <!-- table contents -->
  </table>
</div>
```

```css
.code-scroll,
.table-scroll {
  overflow: auto;
  max-width: 100%;
}

.code-scroll:focus,
.table-scroll:focus {
  outline: 2px solid #0066cc;
  outline-offset: 2px;
}
```

# F8. Iframes

## F8. Iframes

### Checklist
- Every iframe needs a descriptive `title` attribute.
- The title should describe the content or purpose of the iframe.
- If the iframe is decorative or redundant, hide it from assistive technologies when appropriate and do not expose duplicate content.
- Ensure the iframe content itself is accessible.
- Keyboard users must be able to move into and out of the iframe.
- Users should be able to tell that content is embedded.
- Consider whether an iframe is necessary or whether the content could be embedded directly in the page instead.
- The content inside the iframe should follow the same accessibility rules as the rest of the page.
- Lazy-loaded iframes should still provide a meaningful title before loading.

# G1. Animations and Transitions

### Question  
> _What type of animation?_  

### Options  
- CSS transitions (hover, focus, state changes)
- CSS keyframe animations (loading spinners, decorative motion)
- JavaScript-driven animations (scroll animations, page transitions)
- Animated images (GIFs, animated SVGs, animated WebP)

---

## G1-1. CSS Transitions

### Checklist
- Respect the `prefers-reduced-motion` media query.
- Provide reduced motion or no motion for users who prefer it.
- Keep transitions short; around **500ms or less** is a good upper limit for most interface transitions.
- Do not animate properties that cause layout shifts if a simpler effect will work.
- Transitions should not distract from the main content.
- Ensure transitioning elements remain readable during the transition.
- Focus indicators should not have slow transitions; focus should appear immediately or very quickly.
- Do not move content while the user is trying to interact with it.

### Code
```css
button {
  background-color: #0066cc;
  transition: background-color 200ms ease;
}

button:hover {
  background-color: #005199;
}

@media (prefers-reduced-motion: reduce) {
  button {
    transition: none;
  }

  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

- Prefers-reduced-motion: [MDN prefers-reduced-motion](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion)

## G1-2. CSS Keyframe Animations

### Checklist
- Respect the `prefers-reduced-motion` media query.
- Provide a static alternative when motion is reduced.
- Loading spinners should provide a text alternative such as “Loading…”.
- Decorative animations should stop or be removed when reduced motion is requested.
- Do not use infinite animation for content that conveys information unless users can pause or stop it.
- Looping animations that continue for more than **5 seconds** need a pause, stop, or hide mechanism unless an exception applies.
- Ensure animated elements do not flash more than **3 times in any 1 second period**.

### Code
```html
<div class="loading">
  <div class="spinner" aria-hidden="true"></div>
  <p>Loading content, please wait…</p>
</div>

<div aria-live="polite">
  <p>Loading…</p>
</div>
```

## G1-3. JavaScript Animations

### Checklist
- Check `prefers-reduced-motion` in JavaScript before animating.
- Provide a mechanism to pause or stop animation when needed.
- Scroll-triggered animations should respect reduced motion preferences.
- Page transitions should not disorient users.
- Animated content must remain readable while moving.
- Do not animate layout shifts that move content users are trying to read or interact with.
- Keep animations smooth and efficient, for example by using `requestAnimationFrame`.
- Provide another way to access content revealed by animation if the animation is reduced or removed.

## G1-4. Animated Images (GIFs, Animated SVGs, WebP)

### Checklist
- Animated images that continue longer than **5 seconds** need a pause, stop, or hide mechanism unless an exception applies.
- Provide a pause button near the image when the motion persists.
- Consider using video instead of GIF when better playback control is needed.
- Respect `prefers-reduced-motion` where possible, for example by swapping to a static image.
- Animated images must not flash more than **3 times in any 1 second period**.
- Provide alt text describing the content or purpose, not just that it is animated.
- Decide whether the animation is necessary or decorative.
- If it is purely decorative, let users stop or hide it when needed.

### Code
```html

</picture>
```

# G2. Auto-Playing Content

### Question  
> _What type of auto-playing content?_  

### Options  
- Auto-playing video
- Auto-playing audio
- Auto-rotating carousel (see Branch E: Carousels)
- Auto-scrolling text or news ticker

---

## G2-1. Auto-Playing Video

### Checklist
- Strongly consider not auto-playing video at all.
- If a video auto-plays, it should be muted by default.
- Provide clearly visible pause and stop controls.
- The pause control should appear before the video in DOM order when possible.
- If motion starts automatically and lasts more than **5 seconds**, users need a way to pause, stop, or hide it.
- If the video has audio, it should not auto-play with sound, or it must provide compliant audio controls.
- The pause mechanism should be easy to find and use.
- Auto-play should stop when any element inside the component receives keyboard focus.
- Respect `prefers-reduced-motion`.
- Provide captions if the video contains meaningful audio.
- Do not use auto-playing video only as a background behind important text unless readability is preserved.

### Code
```html
<div class="hero">
  <button type="button" id="video-toggle">Pause background video</button>

  <video autoplay loop muted playsinline id="bg-video">
    <source src="background.mp4" type="video/mp4">
  </video>

  <div class="hero-content">
    <h1>Welcome to Our Site</h1>
  </div>
</div>
```

```Javascript
const reduced = window.matchMedia('(prefers-reduced-motion: reduce)');  
const video = document.getElementById('bg-video');  
  
if (reduced.matches) {  
video.pause();  
}  
  
document.getElementById('video-toggle').addEventListener('click', () => {  
if (video.paused) {  
video.play();  
} else {  
video.pause();  
}  
});
```

## G2-2. Auto-Playing Audio

### Checklist
- Do not auto-play audio with sound.
- If audio must auto-play, it should last less than **3 seconds** or provide a mechanism to pause or stop it.
- Provide controls to pause or stop audio within the first **3 seconds**.
- Provide a way to control volume independently of the system volume.
- Ensure the control mechanism is keyboard accessible.
- Make sure the controls appear early in the page order.
- Audio that plays longer than **3 seconds** must have pause or stop or volume controls that meet WCAG requirements.
- Users of screen readers must be able to hear their assistive technology over any audio.

### Code
```html
<div class="audio-controls">
  <button type="button" id="mute-audio">Mute background audio</button>

  <label for="volume">Volume</label>
  <input type="range" id="volume" min="0" max="100" value="50">
</div>

<audio id="bg-audio">
  <source src="background-music.mp3" type="audio/mpeg">
</audio>
```

```javascript
const audio = document.getElementById('bg-audio');
```

## G2-3. Auto-Scrolling Content

### Checklist
- Provide visible pause or stop controls.
- Pausing must not hide the content.
- Stop scrolling when the user hovers over or focuses the content if that supports readability.
- Scrolling content should stop when it receives keyboard focus.
- Provide another way to read the same content, such as a full list or archive view.
- Ensure the content remains readable at the chosen scrolling speed.
- Respect `prefers-reduced-motion`.
- Auto-scrolling content that continues longer than **5 seconds** needs pause, stop, or hide controls unless an exception applies.
- Content should not scroll so fast that it cannot be read.
- Consider whether scrolling is needed at all.

### Code
```html
<div class="ticker-container">
  <button type="button" id="pause-ticker">Pause news ticker</button>
```

# G3. Timed Interactions

### Question  
> _What type of timed interaction?_  

### Options  
- Session timeout (login session expires)
- Form with time limit (exam, booking)
- Countdown timer (sale, event)
- Content that disappears after a set time (toast notifications)

---

## G3-1. Session Timeouts

### Checklist
- Warn users before the session expires.
- Show the warning at least **20 seconds** before timeout.
- Provide an option to extend the session.
- The extension mechanism must be keyboard accessible.
- The extension mechanism should be easy to find and use.
- Let users extend the session at least **10 times**, unless the timing is essential or another WCAG exception applies.
- After extending, reset the timer clearly.
- Announce important timeout warnings to screen readers, for example with `role="alert"` or another appropriate live region.
- Do not log users out without warning.
- Consider offering “remember me” or longer sessions for users who need more time.
- Save user data so progress is not lost if a timeout occurs.

### Code
```html
<div
  role="alertdialog"
  aria-modal="true"
  aria-labelledby="timeout-title"
  aria-describedby="timeout-desc"
  hidden>
  <h2 id="timeout-title">Session expiring soon</h2>
  <p id="timeout-desc">Your session will expire in 2 minutes due to inactivity. Would you like to continue?</p>

  <button type="button" id="extend-session">Continue session</button>
  <button type="button" id="logout">Log out</button>
</div>
```

```javascript
let warningTimer;
let sessionTimeout;

function startSessionTimer() {
  warningTimer = setTimeout(() => {
    showTimeoutDialog();
  }, 28 * 60 * 1000);

  sessionTimeout = setTimeout(() => {
    logoutUser();
  }, 30 * 60 * 1000);
}

function extendSession() {
  clearTimeout(warningTimer);
  clearTimeout(sessionTimeout);
  startSessionTimer();
  hideTimeoutDialog();
}
```

## G3-2. Timed Forms

### Checklist

- Allow at least **10 extensions**, unless an exception applies.
- Save progress regularly so work is not lost.
- Warn when time is running low, for example at least **20 seconds** before expiry.
- Announce time warnings to screen readers.

## G3-3. Countdown Timers

### Checklist
- Timer displays should be visible and clear.
- Do not update a live region every second because that is too noisy for screen reader users.
- Announce significant milestones instead, such as halfway, **5 minutes left**, or **1 minute left**.
- Provide text describing what happens when the timer ends.
- Give users another way to know about the deadline in addition to the ticking timer.
- Ensure timer text has sufficient contrast.
- If the timer triggers an action, warn users before it happens.
- Consider time zone differences for event countdowns.

## G3-4. Toast Notifications

### Checklist
- A toast should remain visible long enough to read, usually at least **5 to 6 seconds** for short messages.
- Do not auto-dismiss critical or error messages; keep them visible until dismissed.
- Success messages may auto-dismiss, but only after enough time to read them.
- Announce toast content to screen readers using `role="status"` or `aria-live="polite"` for non-urgent updates.

- Toasts must not block other important content or interactive elements.
- Toasts should appear in a consistent location.
- Multiple toasts should not overlap in a way that hides content.

# G4. Parallax or Scroll-Based Effects

## G4. Parallax and Scroll Effects

### Checklist
- Respect the `prefers-reduced-motion` media query.
- Provide a static alternative when reduced motion is preferred.
- Do not make content accessible only by scrolling effects.
- Parallax should not make text difficult to read.
- Ensure sufficient contrast between text and any moving background.
- Do not move content so fast that it causes nausea or disorientation.
- Background movement should not distract from the main content.

# G5. Flashing or Blinking Content

## G5. Flashing and Blinking Content

### Checklist
- Content must not flash more than **3 times in any 1 second period**, unless it is below the general and red flash thresholds.
- Flashing content can cause seizures in people with photosensitive epilepsy.
- This includes video, animations, GIFs, and CSS effects.
- If content might flash, test it with the [Photosensitive Epilepsy Analysis Tool (PEAT)](https://trace.umd.edu/peat).
- Avoid large flashing areas; the larger the flashing area, the more dangerous it can be.
- Red flashing is especially dangerous and should be avoided.
- Warn users before showing any potentially dangerous content.
- Let users stop or skip flashing content.
- Do not use CSS blinking text like `text-decoration: blink`.
- Blinking or flashing that lasts more than **5 seconds** also needs pause, stop, or hide controls unless an exception applies.

### Code
```html
<div role="alert">
  <p><strong>Warning:</strong> The following video contains flashing lights and may not be suitable for people with photosensitive epilepsy.</p>
  <button type="button">Continue</button>
  <button type="button">Skip this content</button>
</div>
```

```css
/* Never do this */
.blink {
  text-decoration: blink;
}

@keyframes gentlePulse {
  0%, 100% { opacity: 1; }
  50% { opacity: 0.75; }
}

.attention {
  animation: gentlePulse 2s ease-in-out 1;
}

@media (prefers-reduced-motion: reduce) {
  .attention {
    animation: none;
  }
}
```

# H1. Tab Order and Focus Management

## H1. Tab Order and Focus Management

### Checklist
- DOM order should match visual order.
- Do not use positive `tabindex` values such as `1`, `2`, or `3`.
- Use native interactive elements like `<button>`, `<a>`, `<input>`, and `<select>` whenever possible.
- Only interactive elements should normally be in the Tab order.
- Hidden content should not be focusable. Use `hidden`, `display: none`, or `inert` when appropriate.
- When content is removed, move focus to a logical place.
- When a modal closes, return focus to the element that opened it.
- Use `tabindex="0"` only when a normally non-focusable element must become keyboard focusable for a valid reason.
- Use `tabindex="-1"` only for programmatic focus, not for normal Tab navigation.
- Focus should never disappear to nowhere or land on `<body>` unexpectedly.

### Tabindex Quick Reference
- Not present → default browser behavior
- `0` → focusable in natural DOM order
- `-1` → focusable only through script or manual focus management
- Positive values → avoid; they create unnatural focus order

# H2. Focus Indicators

## H2. Focus Indicators

### Checklist
- All interactive elements must have a visible focus indicator.
- Do not remove the browser focus outline unless you provide an accessible replacement.
- The focus indicator must have at least **3:1 contrast** against adjacent colors.
- The visible focus indicator should be at least **2 CSS pixels thick** or otherwise clearly visible.
- Use `:focus-visible` for keyboard-only focus styling where appropriate.
- Test focus indicators on light, dark, and mixed backgrounds.
- Test in Windows High Contrast Mode or equivalent forced-colors mode.
- The focused component itself or its focus indicator must meet WCAG 2.4.13 focus appearance requirements.

# H3. Keyboard Traps and Modals

## H3. Avoiding Keyboard Traps

### Checklist
- Users must always be able to Tab out of any component unless focus is intentionally trapped inside a modal dialog.
- Do not override the Tab key without providing a clear and reliable escape.
- Arrow keys may be used for navigation inside composite widgets, but Tab should still move in and out as expected.
- Test third-party widgets, embeds, and custom components for keyboard traps.

## H3. Modal Focus Management

### Checklist
- Focus should move into the modal when it opens.
- Tab should cycle only within the modal while it is open.
- Escape should close the modal unless there is a strong reason not to.
- When the modal closes, focus should return to the trigger.
- Background content should be made inert or otherwise removed from interaction while the modal is open.

### Code
```html
<main inert>
  <!-- Background page content when modal is open -->
</main>

  <!-- Modal content -->
</div>
```

# H4. Custom Keyboard Interactions

## H4. Expected Keyboard Behavior

### Typical keyboard behavior
- Link: Tab focuses, Enter activates
- Button: Tab focuses, Enter activates, Space activates
- Checkbox: Tab focuses, Space toggles
- Radio group: Tab enters group, arrow keys move selection, Space or arrow keys select depending on implementation
- Select: Tab focuses, Enter or Space opens, arrow keys navigate, Escape closes
- Tabs: Tab reaches tablist, arrow keys switch tabs, Enter or Space may activate depending on pattern
- Menu button: Tab focuses trigger, Enter or Space opens, arrow keys navigate, Escape closes
- Modal: focus moves inside, Escape closes

## H4. Composite Widgets (tabs, menus, listbox)

### Checklist
- Only one element inside the composite widget should usually be in the Tab order.
- Arrow keys should move between internal items.
- Tab should move to the next component outside the widget.
- Enter or Space should activate the current item when appropriate.

## H4. Custom Buttons and Links

### Checklist
- If you build a custom control, give it `role="button"` or `role="link"` only when you cannot use a native element.
- A custom button or link should have `tabindex="0"` so it can receive focus.
- A custom button must respond to Enter and Space.
- A custom link must respond to Enter.
- Prefer native HTML elements over ARIA whenever possible.

- All component patterns: [ARIA APG - All Component Patterns](https://www.w3.org/WAI/ARIA/apg/patterns/)
- ARIA button role: [MDN - ARIA button role](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Roles/button_role)

# H5. Shortcut Keys

## H5. Shortcut Keys

### Checklist
- Avoid single-character shortcuts unless users can turn them off, remap them, or use them only when focus is in a specific component.
- Use modifier keys for global shortcuts, such as Ctrl+K, not just K.
- Do not conflict with browser shortcuts.
- Do not conflict with screen reader shortcuts.
- Document all keyboard shortcuts for users.
- Access keys (`accesskey`) often conflict with assistive technology and browser commands, so use them sparingly or avoid them.

### Learn More
- Character key shortcuts: [WCAG 2.1.4 - Character Key Shortcuts](https://www.w3.org/WAI/WCAG22/quickref/#character-key-shortcuts)
- accesskey attribute: [MDN - accesskey attribute](https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/accesskey)
- Keyboard accessibility: [AIM - Keyboard Accessibility](https://webaim.org/techniques/keyboard/)

# H6. Skip Links

## H6. Skip Links

### Checklist
- The skip link should be the first focusable element on the page.
- It should become visible when focused.
- It should point to the main content target, usually `#main`.
- The target should be focusable when needed, often with `tabindex="-1"` on the target element.
- Skip link text should clearly describe what it skips to, such as “Skip to main content”.
- The skip link must work with keyboard navigation.
- The skip link text and focus state must be clearly visible.

### Learn More
- Bypass blocks: [WCAG 2.4.1 - Bypass Blocks](https://www.w3.org/WAI/WCAG22/quickref/#bypass-blocks)
- Skip navigation links: [AIM - Skip Navigation Links](https://webaim.org/techniques/skipnav/)

# H7. Testing Keyboard Accessibility

## H7. Quick Test Checklist

Tab through the page and verify:
- Every interactive element is reachable with Tab.
- Focus indicators are visible on every focused element.
- Tab order matches the visual and logical order.
- There are no keyboard traps.
- Enter activates links and buttons.
- Space activates buttons and toggles where appropriate.
- Escape closes modals, menus, and popups where expected.
- Arrow keys work inside composite widgets such as menus, tabs, listboxes, and radio groups.
- Skip links work.
- Focus returns appropriately after closing overlays or dialogs.
